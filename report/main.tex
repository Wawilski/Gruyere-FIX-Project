\section{Introduction}

In this homework, we were asked to search and fix 4 different exploits seen in the Google Gruyere website. During this report, we will explain the different exploits, and propose a fix for each of them.

\section{Cookie Manipulation}
In this section, the cookie manipulation exploit is addressed. 
This exploit consists of using cookies logging into another user account
without their password. It also allows logging with admin privileges as non-privileged user (which is a kind of elevation
of privilege).\\
\S\ref{CookieExpl} explains the exploit and how to recreate it. \\
\S\ref{CookieFix} explains how we addressed the exploit and fixed it. \\
\S\ref{CookieValidation} validates the absence of the exploit after the implemented fix. 

\subsection{Exploit}\label{CookieExpl}

To replicate this exploit, let's take a user named \code{new}. The user cookie
of \code{new} is illustrated in Figure \ref{cookie_new}. The format of the
cookie follows this pattern: \code{hash | User ID | is\_admin | is\_author},
where is\_admin (resp. is\_author) defines if the user is an admin (resp.
author).

\begin{figure}[H]
  \centering 
  \includegraphics[width = 0.9\textwidth]{images/cookie_new.png} 
  \caption{Cookie of the first user.  }
  \label{cookie_new}
\end{figure}

Let's create a new user with the name \code{new|admin|author}, this user
cookie is showed in Figure \ref{cookie_newAdmin}. 

\begin{figure}[H] 
  \centering 
  \includegraphics[width =0.9\textwidth]{images/cookie_exploit.png}
  \caption{Cookie of the new user }
  \label{cookie_newAdmin}
\end{figure}

Here, if we want to log into the second account we will be logged into the
first account with admin privileges, and the true privileges of the user
are ignored.

\begin{figure}[H]
  \centering 
  \includegraphics[width =0.9\textwidth]{images/cookie_admin.png} 
  \caption{User logged into the first account with admin privileges}
\end{figure}

\subsection{Fix}\label{CookieFix}

The issue here is that, in the code, we dont separe the UID and the other
fields. When we parse the cookie, we split the fields via the pipes in the 
cookie's value, so if we put pipes in our User ID it will split the ID too. 
After the split we check the fields from left to right, so if we put the right
fields in the User ID we can are able to create a made-up cookie value inside
the true cookie, and so log into another account and choose the level of
privileges wanted. 

\begin{figure}[H]
        \begin{minted}[
            % frame=lines,
            % framesep=2mm,
            % baselinestretch=1.2,
            bgcolor=codebg,
            fontsize=\footnotesize,
            % linenos
            ]{java}

  def _ParseCookie(self, cookie):
      """Parses the cookie and returns NULL_COOKIE if it's invalid.

      Args:
        cookie: The text of the cookie.

      Returns:
        A map containing the values in the cookie.
      """
      try:
          hashed, cookie_data = cookie.split("|", 1)
          # global cookie_secret
          if hashed != str(hash(cookie_secret + cookie_data) & 0x7FFFFFF):
              return self.NULL_COOKIE
          values = cookie_data.split("|")
          return {
              COOKIE_UID: values[0],
              COOKIE_ADMIN: values[1] == 'admin',
              COOKIE_AUTHOR: values[2] == 'author',
          }
      except (IndexError, ValueError):
          return self.NULL_COOKIE

        \end{minted}
    \caption{initial Cookie Parsing}\label{fig:initCookieParsing}
\end{figure}


A possible fix here, is to take the name appart of the other fields.
As we know that the cookie will create the last two fields by
default we can check these two in priority after the split and concatenate 
the rest of the cookie as the User ID.

All the changes regarding the fix of the Cookie Manipulation vulnerability is
assessed in
\href{https://github.com/Wawilski/Gruyere-FIX-Project/commit/541b9429f8d6e9deca442883aea5a0b329dfac99}{this
commit}.

\begin{figure}[H]
        \begin{minted}[
            % frame=lines,
            % framesep=2mm,
            % baselinestretch=1.2,
            bgcolor=codebg,
            fontsize=\footnotesize,
            % linenos
            ]{java}

  def _ParseCookie(self, cookie):
      """Parses the cookie and returns NULL_COOKIE if it's invalid.

      Args:
        cookie: The text of the cookie.

      Returns:
        A map containing the values in the cookie.
      """
      try:
          hashed, cookie_data = cookie.split("|", 1)
          # global cookie_secret
          if hashed != str(hash(cookie_secret + cookie_data) & 0x7FFFFFF):
              return self.NULL_COOKIE
          values = cookie_data.split("|")
          return {
              COOKIE_UID: "|".join(values[0:-2]),
              COOKIE_ADMIN: values[-2] == "admin",
              COOKIE_AUTHOR: values[-1] == "author",
          }
      except (IndexError, ValueError):
          return self.NULL_COOKIE

        \end{minted}
    \caption{fixed Cookie Parsing}\label{fig:CookieParsing}
\end{figure}


\subsection{Validation}\label{CookieValidation}

Now, if we log into the \code{new|admin|author} account, we won't access to
the \code{new} account (even if the cookie stays the same as before).

\section{Elevation of Privilege}

In this section, the elevation of privilege is addressed. 
This issue consists of giving a user privileges higher than his current ones 
(in this case admin privileges).\\
\S\ref{ElevationExpl} explains the exploit and how to recreate it. \\
\S\ref{ElevationFix} explains how we addressed the exploit and fixed it. \\
\S\ref{ElevationValidation} validates the absence of the exploit after the implemented fix. 

\subsection{Exploit}\label{ElevationExpl}

When we create or change a profile, we use the \code{/saveprofile} url, with
the fields needed to create or change the profile.

Here, using \code{is\_admin=True} in the url's field, we can create an admin
user or change the privileges of an existing user to admin, without any
verification. 

So, these url templates (with a chosen \code{user\_uid}) can be used, for respectively create a user admin and change a user privileges to admin: 

\begin{itemize}
  \item \code{/saveprofile?action=new\&is\_admin=True}\\
    \code{\&is\_author=True\&uid=user\_uid\&pwd=user\_pwd}
  \item \code{/saveprofile?action=update\&is\_admin=True}\\
    \code{\&uid=user\_uid}
\end{itemize}

\subsection{Fix}\label{ElevationFix}

The issue here is that there is no verification about any permission 
when we create a new user or change a user profile. A possible fix here is
to only allow admins to create other users.

assessed in
\href{https://github.com/Wawilski/Gruyere-FIX-Project/commit/9724b92b281e21da648befeea0d0e2b33523cb8b}{this
commit}.

\section{Path Traversal}
In this section, the path traversal attacks are addressed.
Path traversal attacks consist of accessing files outside the original scope of the website by using a malicious path. 
This exploit can lead to devastating information leak. For example, if the server is running with root privileges, it might be possible to access the \code{/etc/shadow} file.
Path traversal can also be used to upload malicious files in directory outside the website context.\\
\S\ref{pathTraversalExpl} explains the exploit and how to recreate it. \\
\S\ref{pathTraversalFix} explains how we addressed the exploit and fixed it. \\
\S\ref{pathTraversalValidation} validates the absence of the exploit after the implemented fix. 
\subsection{Exploit}\label{pathTraversalExpl}
The exploit is as simple as using \code{../} to go back in the context path.
Tough to be able to enter the \code{/} we have to use \code{\%2f} tag for it to be considered. 

Figure \ref{fig:pathTraversalExpl} describes how to exploit the path traversal vulnerability. It demonstrates how an innocent file (\code{secret.txt} from the Gruyère app) can be accessed as well as more sensible files  (\code{/etc/passwd} file). 

\begin{figure}[ht]
    \begin{minted}{bash}
$ curl http://127.0.0.1:8008/..%2f/secret.txt

$ curl http://127.0.0.1:8008/..%2f/..%2f/..%2f/..%2f/..%2f/..%2f/etc/passwd

git:x:973:973:git daemon user:/:/usr/bin/git-shell
....
    \end{minted}
    \caption{Example of the path traversal attack.}\label{fig:pathTraversalExpl}
\end{figure}

\subsection{Fix}\label{pathTraversalFix}
To fix the path traversal vulnerability, we figured it was necessary to only accept a restricted list of urls and reject every request not matching any of these urls.

To this end, we initialize a list of authorized urls to be the empty url (\code{/}), the pages defined by the \code{.gtl} files and the methods such as login, reset, etc (see Figure \ref{fig:PathtraversalAuthorized}).  
Then, to be able to consider uploaded files, in the \code{HandleRequest(self, path, query, unique\_id)} method, we crawl in the ressource folder to find if the requested url correspond to added content (see Figure \ref{fig:PathtraversalCheck}).

All the changes regarding the fix of the Path traversal vulnerability is assessed in \href{https://github.com/Wawilski/Gruyere-FIX-Project/commit/27f2a1523d54b69ea82b231bbb409345ae083d1e}{this commit}.

    # crawls the ressources folder for .gtl files
    def _getValidUrls():
        # starts with the empty url
        urls = ["/"]

        )

    # The only urls that are accepted (to avoid path traversal)

\section{Denial of Service}
In this section, the Denial of Service is addressed.
Denial of service attacks aim at taking down the proper functionning of a device or service. This leads to the violation of the availability porperty from th CIA triad (Confidentiality, Integrity, and Availability). It is one of the most common attacks on the internet.\\
\S\ref{sec:DoSExpl} explains different Denial of Service exploits and how to recreate them. \\
\S\ref{sec:DoSFix} explains how we addressed the exploits and fixed them. \\
\S\ref{sec:DoSValidation} validates the absence of the exploits after the implemented fixes. 

\subsection{Exploits}\label{sec:DoSExpl} 
\subsubsection{Quitting the server}
To shut down the server, administrator privileges should be required. But as represented in Figure \ref{fig:DoSquit}, simply visiting the \code{/quitserver} url is sufficient to shut it down without any prior login or cookie.

\subsection{Fixes}\label{sec:DoSFix} 
\subsubsection{Quitting the server}
The fix of this vulnerability is as simple as a single line of code. As the \code{\_PROTECTED\_URLS} variable is set to  
\code{["/quit", "/reset"]} which, to protect the \code{quitserver} url, as to be changed to \code{["/quitserver", "/reset"]}.
\href{https://github.com/Wawilski/Gruyere-FIX-Project/commit/a897c89d98a331abe22039d7f05f2c4d8d041d48}{This commit} addresses this change.


\subsubsection{Resetting the server}
This vulnerability is already addressed by the Path Traversal fix, as the \code{/RESET} url isn't a correct url.
